# Практическая работа 1 Анализ требований к приложению.

## Вариант 15: 	Панель администратора для управления пользователями.

## Задание
Проанализировать требования к системе авторизации пользователей по логину и паролю, определить виды требований (бизнес-, функциональные, нефункциональные), сформировать пользовательскую историю, критерии приемки, выявить возможные неясности или риски, и оформить результаты в файл README.md, подготовить репозиторий.

### Бизнес-требования
1.Обеспечить администраторам централизованный контроль над пользователями системы.
2.Упростить процессы модерации, блокировки и восстановления доступа пользователей.
3.Повысить безопасность и соответствие политикам доступа.
### Функциональные требования
Администратор может просматривать список всех пользователей с фильтрацией и сортировкой.
Администратор может искать пользователей по логину, email или ID.
Администратор может блокировать/разблокировать учетные записи пользователей.
Администратор может сбрасывать пароли пользователей и отправлять уведомления об этом.
Администратор может просматривать историю действий пользователей (логи входа, изменения данных).
Администратор может назначать и изменять роли пользователей (например, модератор, пользователь, администратор).
### Нефункциональные требования
Время загрузки списка пользователей не должно превышать 3 секунд.

Доступ к панели администратора разрешен только авторизованным пользователям с ролью администратора.

Все действия администратора должны логироваться для аудита.

Интерфейс должен быть интуитивно понятным и адаптированным для работы с большими объемами данных.

Система должна поддерживать одновременную работу нескольких администраторов.

## User Story
Как администратор системы, я хочу иметь возможность управлять учетными записями пользователей, чтобы обеспечивать безопасность и порядок в системе.
## Acceptance Criteria
1.При вводе правильных логина и пароля пользователь входит в систему и перенаправляется в личный кабинет.
2.При неправильных данных отображается сообщение с просьбой повторить.
3.После 3 неуспешных попыток входа — временная блокировка на 5 минут и сообщение об этом.
4.Пользователь может запросить восстановление пароля через свою электронную почту.

## Возможные неясности / риски
1.Можно ли использовать только логин или обязательно использовать электронную почту для входа?
2.Какой минимальный и максимальный размер логина и пароля?
3.Требуется ли внедрение CAPTCHA после нескольких неправильных попыток входа?
4.Планируется ли хранение историю входов и их статус?

Это позволило структурировать требования и подготовить их к дальнейшей реализации.
